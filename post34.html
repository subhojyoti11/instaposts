Why not just use a sign bit?

The obvious approach: reserve the leftmost bit as a sign flag. 0 = positive, 1 = negative. This is called sign-magnitude. Early computers used it. It failed.

Sign-magnitude (the naive approach):
────────────────────────────
+5 = 00000101
-5 = 10000101 ← just flip sign bit
────────────────────────────
Problem 1: TWO ZEROS
+0 = 00000000
-0 = 10000000 ← what is negative zero??
────────────────────────────
Problem 2: ADDITION BREAKS
0101 (+5)
+1011 (-3)
=10000 ← this is -0, not +2!
────────────────────────────
The CPU needs SEPARATE circuits for addition and subtraction.
Doubles the hardware. Doubles the cost. Doubles bugs.

Sign-magnitude failed because math doesn't work. You can't just add two sign-magnitude numbers with a simple adder circuit. You need to check signs first, compare magnitudes, then decide whether to add or subtract. Expensive, slow, error-prone.

Flip every bit.
Add one.

To negate any number in two's complement: invert all bits, then add 1. That's the entire algorithm.

Convert +5 to -5 (8-bit):
────────────────────────────
Step 1: Start with +5
0000 0101
────────────────────────────
Step 2: Flip every bit
1111 1010
────────────────────────────
Step 3: Add 1
1111 1011 ← this is -5
────────────────────────────
Verify: -128+64+32+16+8+0+2+1 = -128 + 123 = -5 ✓

The leftmost bit has weight -128 (in 8-bit), not just a sign flag. It actively contributes a negative value. That's why the math works automatically. 1111 1011 isn't "negative 0000 0101." It's a completely different number where the MSB contributes -128.

Only ONE zero: 0000 0000 = 0. Flip all bits = 1111 1111. Add 1 = 1 0000 0000. The overflow bit is discarded. Result: 0000 0000 = 0 again. One zero. Problem solved.

Subtraction doesn't exist.

In two's complement, 5 - 3 is really 5 + (-3). The CPU only needs an adder circuit. No subtractor. One circuit does everything.

Compute: 5 - 3 = ?
────────────────────────────
CPU converts to: 5 + (-3)
────────────────────────────
0000 0101 (+5)
+ 1111 1101 (-3 in two's complement)
────────────────────────────
= 0000 0010 (+2) ✓
────────────────────────────
The carry-out overflows and is discarded.
No sign checking. No magnitude comparison.
No special cases. Just addition.

This is why two's complement won. One adder circuit handles positive + positive, positive + negative, negative + positive, and negative + negative. The hardware doesn't even know if the numbers are signed or unsigned. The same bits, the same circuit, correct result either way.

You use it every day.

int
Every int in C, Java, Rust, Go, Swift is a 32-bit two's complement number. Range: -2,147,483,648 to +2,147,483,647. The negative side has one extra number
MIN
Integer.MIN_VALUE in Java is -2,147,483,648. Negate it and you get... -2,147,483,648 again. Because +2,147,483,648 doesn't fit in 32 bits. The "weird number"
abs()
Math.abs(Integer.MIN_VALUE) returns a NEGATIVE number. This is a real bug in production systems. Two's complement is why
x86
Your CPU's ADD instruction works for both signed and unsigned integers. Same circuit, same opcode. It literally cannot tell the difference. Two's complement makes this possible
-x
The unary minus operator -x in every language compiles to NOT(x) + 1. Flip all bits, add one. The compiler emits two's complement, not "make it negative"

One circuit to rule them all.

1
One adder circuit. Handles addition AND subtraction. No separate subtractor needed. Saves billions of transistors on every chip ever manufactured
2
One zero. No +0 and -0 confusion. Comparison is a single operation. Branching logic is simpler
3
No sign checking. The CPU doesn't check if a number is positive or negative before adding. It just adds. The result is correct regardless
4
Overflow detection. One XOR of the two most significant carry bits. If they differ, overflow occurred. Simple, fast, one gate

Two's complement has been the standard since the 1960s. Every x86, ARM, RISC-V, and MIPS processor uses it. Every programming language assumes it. Every signed integer on every device on Earth is stored this way. It's so fundamental that most engineers forget it's even a design choice.

The lesson: The best engineering decisions are invisible. Two's complement is so good that 60 years later, nobody has found anything better. Flip the bits, add one. That's it. That's the whole trick that runs the modern world.
