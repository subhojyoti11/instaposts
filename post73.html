The Cold Start
Problem.

What happens
When no instance of your Lambda function is already running, AWS must provision a new execution environment from scratch.
The numbers
Cold start latency varies massively by runtime. Go: 50-100ms. Node.js: 100-250ms. Python: 100-300ms. Java with Spring: 2,000-10,000ms.
When it happens
Cold starts occur when a function receives its first invocation, after it scales up to handle a traffic spike, or after 15 minutes of idle time.
User impact
A cold start adds imperceptible 50ms for a Go function and catastrophic 8 seconds for a Spring Boot service.

03 / 07

What Happens
Inside a
Cold Start.

Init 1
Execution environment provisioning. AWS allocates a Firecracker microVM, loads the Lambda worker process, and mounts the container image from the Lambda service layer.
Init 2
Runtime startup. Python loads, Node.js V8 starts, or the JVM launches its bytecode verifier and JIT compiler.
Init 3
Handler initialization code runs. Your imports, global variables, SDK clients, and database connection pools all initialize once per cold start.
Warm
Subsequent requests reuse the same execution environment. AWS keeps the microVM alive for up to 15 minutes of idle time, making warm invocations 10-100x faster.

05 / 07

SnapStart and
Mitigation.

SnapStart
AWS Lambda SnapStart publishes a snapshot of the initialized JVM execution environment. Subsequent cold starts restore from snapshot, dropping Java from 8s to under 1s.
Provisioned concurrency
Pre-warms N execution environments continuously so they are always ready. Eliminates cold starts but costs ~2x on-demand Lambda pricing.
Optimize the runtime
Switch from Java/Spring to GraalVM native image to eliminate JVM startup. GraalVM native drops Spring Boot cold starts from 8,000ms to 120ms.
Lazy loading
Defer heavy imports inside the handler function, not at module level. Cache SDK clients and DB connections as globals after first use.

In
Production.

Figma
Migrated multiplayer server from Node.js Lambda to a long-running Rust service. Cold start variance made real-time WebSocket collaboration unreliable.
Datadog
Built a Lambda layer that tracks cold start frequency and duration across customer functions. Telemetry shows Java cold-starting on 1 in 100 requests, dominating p99 latency.
Netflix
Uses Provisioned Concurrency on Lambda functions in the critical path for stream startup. Cold start variance in the first 3 seconds of playback directly hurts retention.
Shopify
Adopted WebAssembly for Shopify Functions instead of Lambda. Wasm on Cloudflare Workers cold-starts under 1ms vs 200ms Lambda cold starts.

07 / 07

Quick
Reference.

Cold start
Latency added when Lambda must provision a new execution environment before handling a request. Includes microVM creation, runtime startup, and handler init code.
Warm start
Lambda reuses an existing execution environment, skipping all init phases. Module-level globals persist for connection reuse. Environments live up to 15 min idle.
SnapStart
AWS feature for Java Lambda that snapshots the initialized JVM state after deployment. Reduces Java cold starts from 8,000ms to under 1,000ms without code changes.
Provisioned concurrency
Pre-warms N execution environments so requests always land on a warm instance. Costs ~2x on-demand pricing. Eliminates cold starts entirely.
Firecracker
The microVM hypervisor powering AWS Lambda and AWS Fargate. Open-sourced by AWS in 2018. Boots a minimal kernel in under 125ms with VM-level isolation.

Caption
Serverless promises you only pay for what you use. The cold start is the thing you pay for that you did not ask for. When an AWS Lambda function receives a request and no warm execution environment exists, the service must provision a new one from scratch. A Firecracker microVM boots. The language runtime initializes. Your handler's imports and global initializers run. Only then does your actual code execute. None of this appears in your CloudWatch duration metric. The extra latency is invisible to your application but very visible to your users. The numbers vary dramatically by runtime. Go and Rust cold-start in 50-100ms. Node.js takes 100-250ms. Python is similar. Java with Spring Boot takes 2,000-10,000ms. The JVM's class loading, bytecode verification, and dependency injection framework initialization are catastrophically ill-suited to an environment that may have been idle for 15 minutes. The underlying infrastructure is Firecracker, the open-source microVM hypervisor AWS published in 2018. It powers both Lambda and Fargate. A Firecracker VM boots in under 125ms, which is why Lambda can provision environments at the speed it does. Subsequent requests reuse the same microVM until it has been idle for roughly 15 minutes. AWS SnapStart addresses the Java problem by snapshotting the initialized JVM state to S3 after deployment. Cold starts restore from that snapshot instead of re-initializing, dropping from 8 seconds to under 1 second without code changes. For teams that cannot move off Java, it is the most impactful option. For latency-sensitive critical paths, Provisioned Concurrency pre-warms N execution environments continuously. Netflix uses this for stream startup functions where cold start variance in the first 3 seconds of playback is unacceptable. Shopify avoided the problem entirely by moving checkout functions to Wasm on Cloudflare Workers, where cold starts are under 1ms.
