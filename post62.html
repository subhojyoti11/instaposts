What IaC
Fixed.

Before
Infrastructure provisioned manually through cloud consoles or shell scripts. No version control, no review process, no reproducibility between environments.
Terraform
Declare desired state in HCL. Plan to preview. Apply to make it real. Infrastructure becomes reviewable, diffable, and auditable like any other codebase.
Drift detect
Terraform plan shows what changed between declared state and actual cloud state. Manual console changes are visible and reversible through the plan workflow.
Adoption
Over 1 million organizations use Terraform today. Infrastructure as Code is now the industry baseline, not a best practice for advanced teams.

03 / 07

What HCL
Cannot Do.

No loops
count and for_each exist but break under complex iteration. Creating 20 resources with varying policies requires workarounds that become unreadable fast.
No functions
HCL has built-in functions but no user-defined ones. Reusable logic lives in modules, which cannot be composed or versioned like real software libraries.
No types
HCL is loosely typed. Type errors surface at apply time, not at write time. You find out infrastructure is wrong when it is already half-provisioned in your cloud account.
No tests
No standard way to unit test Terraform modules. Testing means running against a real cloud account. Slow, expensive, and leaves orphaned resources when tests fail.

Mock the Pulumi runtime and verify infrastructure logic without touching AWS. Test before you deploy. Catch mistakes in CI, not in production accounts.
Type safety

07 / 07

Quick
Reference.

IaC
Infrastructure as Code. Declaring desired cloud state in version-controlled files. Terraform, Pulumi, CDK, CloudFormation, Bicep are the major tools.
Terraform
HashiCorp's declarative IaC tool. HCL-based, plan/apply workflow, stateful. The industry standard for multi-cloud provisioning at most organizations.
Pulumi
Infrastructure as Software. Write real Python, TypeScript, or Go to provision cloud resources. Unit-testable, type-safe, composable with real package managers.
AWS CDK
Cloud Development Kit. TypeScript or Python constructs that synthesize to CloudFormation. Typed, composable components for AWS-native infrastructure patterns.
Drift
Gap between declared state and actual cloud state. Detected by Terraform plan, ArgoCD diff, or Pulumi preview. The core problem IaC tools were built to prevent.

Caption
You described your infrastructure in a configuration file. The file became the infrastructure. Then you needed to write a for loop. Terraform changed how the industry thought about cloud. Instead of clicking through a console or running shell scripts, you declared what you wanted and Terraform made it real. Write HCL, run plan, review the diff, apply. The concept was clean. The execution introduced a new class of problems. HCL is not a programming language. It has limited loops, no user-defined functions, no type system, no unit tests. When you need 20 S3 buckets with slightly different policies, you write 20 near-identical blocks or fight with count and for_each until the error messages become unintelligible. When infrastructure needs conditional logic based on environment or region, you stack ternary expressions inside HCL strings. When something breaks, you debug by staring at a 50,000-line plan output with no stack trace. Pulumi took a different position. Infrastructure is a software problem. Solve it with software. Write actual Python, TypeScript, or Go. Use loops, functions, classes, and composition the same way you would for any engineering problem. Write unit tests that mock cloud APIs and verify infrastructure logic before touching a real account. Get type errors at compile time, not at apply time. AWS CDK applied the same idea to CloudFormation. Instead of YAML, you write TypeScript that synthesizes to CloudFormation templates. The AWS team ships constructs, typed components for common infrastructure patterns that compose the way React components do. The underlying shift is that declarative configuration works well until your infrastructure is dynamic. When modules need real logic, when infrastructure has the same complexity as the applications running on it, you need a real language. YAML does not have a debugger. Python does.
