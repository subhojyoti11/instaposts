You add more servers. But now you need a load balancer ; something sitting in front that decides which server gets each request.

The algorithm it uses determines everything: speed, fairness, fault tolerance, and whether adding a server breaks your cache.

03 / 10● ALGORITHM 01

Round Robin (take turns)

The simplest algorithm that actually works.

Requests go to servers in a fixed cycle: S1 → S2 → S3 → S1 → S2 → S3… Dead simple. Zero state to track.

req1 → S1
req2 → S2
req3 → S3
req4 → S1 ← cycle repeats
req5 → S2

Pro: Zero overhead, perfectly fair distribution.
Con: Ignores server health and capacity. A slow server gets the same traffic as a fast one.

// O(1), but naive swipe →
04 / 10● ALGORITHM 02

Weighted Round Robin

Not all servers are created equal.

Same cycle idea, but bigger servers get more turns. A server with weight 3 gets 3× the requests of a server with weight 1.

weights: S1=3 S2=1 S3=2
────────────────────────
req1,2,3 → S1 (weight 3)
req4 → S2 (weight 1)
req5,6 → S3 (weight 2)
→ cycle repeats every 6 requests

Pro: Respects heterogeneous hardware.
Con: Weights are static, doesn't adapt to real-time load.

// capacity-aware, but still blind swipe →
05 / 10● ALGORITHM 03

Least Connections (help the idle one)

Stop guessing. Check who's actually free.

Each new request goes to the server with the fewest active connections right now. It's dynamic, reactive, and adapts in real-time.

active connections:
S1=14 S2=3 S3=11
────────────────────────
new req → S2 ✓ lowest

now: S1=14 S2=4 S3=11
next req → S2 ✓ still lowest

Pro: Adapts to slow servers, long-lived connections.
Con: Needs state tracking. Slightly more overhead per request.

// dynamic, but stateful swipe →
06 / 10● ALGORITHM 04

IP Hash (sticky sessions)

Same user, same server. Every time.

Hash the client's IP address and mod by server count. The same IP always lands on the same server, great for session affinity and local caching.

hash("192.168.1.42") → 7829
7829 % 3 servers = S2
────────────────────────
same user, 100 requests later:
hash("192.168.1.42") → 7829
7829 % 3 = S2 ✓ always S2

Pro: Session persistence without shared session store.
Con: Add/remove a server → every IP remaps. Cache stampede.

// deterministic, but fragile swipe →
07 / 10● ALGORITHM 05

Consistent Hashing (the elegant one)

Add or remove servers without reshuffling the world.

Place servers on a virtual ring (0 to 2³²). Hash each request key and walk clockwise to the nearest server. When a server joins or leaves, only its neighbors are affected.

the ring: 0 ──────── 2³²
S1@pos 1000 S2@pos 4500 S3@pos 8200
────────────────────────
key "user:42" → hash = 3700
walk clockwise → S2 ✓

+ add S4@pos 6000
only keys 4501–6000 move. rest untouched.

Pro: Minimal disruption on scale events.
Con: Uneven distribution without virtual nodes.

// O(log n) and elegant swipe →
08 / 10● ALGORITHMS 06+

The rest of the family

These round out the toolkit each with a sharp trade-off:

06
RandomPick a server at random. Surprisingly effective at scale, no coordination needed. Used in microservices mesh.
07
Least Response TimeRoute to whichever server answered fastest recently. Combines least-connections with latency awareness.
08
Resource-Based (Adaptive)Server reports its own CPU/memory. LB routes to the healthiest node. Heavy monitoring overhead.
09
Power of Two ChoicesPick 2 random servers, send to the one with fewer connections. Near-optimal with almost zero overhead.

09 / 10● IN THE WILD

Where they actually run

Every app you use hits a load balancer. Here's who uses what:

NGINXRound robin + weighted + least connections. The default for most of the web.
AWS ALB / ELBLeast outstanding requests + round robin. Scales to millions of connections.
Cassandra & DynamoDBConsistent hashing as the core partitioning strategy for distributed data.
DiscordConsistent hashing to route millions of guild connections to gateway servers.
CloudflarePower-of-two + least connections at the edge for global CDN routing.
Redis ClusterConsistent hashing to shard cache keys across nodes without reshuffling.

10 / 10● CHEATSHEET

Pick your algorithm

Match the algorithm to your actual constraint:

Simple & stateless

  • Round Robin — equal servers, no tracking
  • Weighted RR — mixed hardware
  • Random — microservices, no coordination
  • IP Hash — session affinity needed

Smart & adaptive

  • Least Conn — long-lived connections
  • Least Response — latency-critical
  • Consistent Hash — caches, sharding
  • Power of Two — best cost/benefit

Start with Round Robin. Upgrade when you hit a real bottleneck. Consistent hashing is the endgame for distributed caches.
