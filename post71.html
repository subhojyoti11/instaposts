What Wasm
Actually Is.

Not JavaScript
WebAssembly is a binary instruction format for a stack-based virtual machine. It is a compilation target, not a language. You write C, C++, Rust, or Go and compile to a .wasm file that runs at near-native speed.
The runtime
A Wasm module runs inside a deterministic, sandboxed virtual machine. The sandbox has no default access to the filesystem, network, or system calls. Capabilities must be explicitly granted through WASI, the WebAssembly System Interface.
WASI
WebAssembly System Interface defines a standard set of POSIX-like syscalls for Wasm modules outside the browser. File access, networking, and clocks are all opt-in capabilities. A module that does not need the network cannot touch it.
Speed
Wasm compiles ahead-of-time via Wasmtime, WasmEdge, or V8 and runs at 70-90% of native C speed. Startup time is microseconds, not the hundreds of milliseconds a container cold start takes. This gap matters enormously at the edge.

03 / 07

Wasm vs
Docker.

Cold start
Docker: 50-500ms to pull image layers, start the daemon, and run an init process. Wasm: sub-millisecond. The Wasm binary loads directly into the runtime with no OS process overhead, no image layers, no init system.
Image size
A minimal Docker image is 5-50MB. A compiled Wasm module for the same function is typically 1-3MB. No OS, no package manager, no libc layers. Just the compiled application code and its dependencies.
Portability
Docker images are architecture-specific: linux/amd64 or linux/arm64. A .wasm binary runs identically on x86, ARM, RISC-V, or any platform with a Wasm runtime. One build, truly every target.
Security
Docker's sandbox is at the OS level, sharing the host kernel. Wasm's sandbox is at the instruction level. A compromised Wasm module cannot call any syscall not explicitly granted. The attack surface is dramatically smaller.

Where Wasm
Wins.

Edge functions
Cloudflare Workers and Fastly Compute run Wasm at 300+ locations worldwide. A request is handled within 10ms of the user at the nearest PoP. Traditional cloud functions in us-east-1 add 100-300ms of round-trip latency.
Plugin systems
Envoy and Kong run Wasm plugins inside the proxy process for zero-copy request filtering. Users upload a Wasm module that intercepts HTTP traffic. The sandbox prevents any plugin from accessing other tenants' data on the same proxy.
Embedded / IoT
WasmEdge targets microcontrollers and constrained devices. A Wasm module for edge inference or protocol parsing runs identically from a Raspberry Pi to a server without recompilation. The runtime footprint is under 1MB.
Serverless cold starts
Wasm component model separates compile time from instantiation time. Pre-compiled Wasm modules instantiate in under 1ms. AWS Lambda SnapStart solves cold starts for JVM functions with 10x more operational complexity than a Wasm runtime achieves natively.

Quick
Reference.

Wasm
Binary instruction format for a portable, sandboxed virtual machine. Compiles from C, C++, Rust, and Go. Runs at 70-90% of native C speed. One binary runs on any platform with a Wasm runtime.
WASI
WebAssembly System Interface: standard syscall API for Wasm outside the browser. Filesystem, network, and clock access are opt-in capabilities. A module that does not request network access cannot make network calls.
Wasmtime
Bytecode Alliance's production Wasm runtime, written in Rust. Used by Fastly Compute and Fermyon Spin. Ahead-of-time compiles Wasm to native code before the request arrives. Module instantiation under 1ms.
Component model
Wasm component model separates compilation from instantiation. Modules declare explicit interfaces using WIT (Wasm Interface Types). Enables composition of Wasm modules from different languages without shared memory risks.
vs Docker
Wasm wins on startup time (microseconds vs hundreds of ms), image size (1MB vs 50MB+), and sandbox isolation. Docker wins on ecosystem maturity, tooling, and support for long-running stateful processes that need full OS access.
