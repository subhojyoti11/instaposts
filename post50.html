What Is the
KV Cache?

K + V
Every input token produces a key and value vector at each attention layer. These are the KV pairs.
Why cache
Recomputing all past KV pairs on every new token is O(n²) in time. The cache makes it O(n).
Grows fast
KV cache size = seq_len × layers × 2 × d_head. A single 13B model request can consume gigabytes.
Bottleneck
At batch size 100, the KV cache, not the model weights, becomes the primary GPU memory consumer.

The Memory
Waste Problem.

# Standard KV cache allocation
max_seq_len = 2048 # pre-allocated
actual_avg = 256 # typical request
waste = 87.5% # per request

# At 100 concurrent requests:
reserved = 100 × 2048 = 204,800 slots
used = 100 × 256 = 25,600 slots
# GPU is 87.5% idle on KV memory

Pre-allocation forces worst-case sizing. You reserve for the longest possible sequence, whether or not you use it. Fragmentation makes it worse.

04 / 07

Borrowed From
the OS.

OS pages
Virtual memory splits RAM into fixed 4KB blocks. Process sees contiguous space; physical layout is scattered.
Page table
Maps virtual addresses → physical pages. The OS manages placement; the process never knows.
KV blocks
PagedAttention splits the KV cache into fixed-size blocks. Each block holds a fixed number of token KV pairs.
Block table
Maps sequence positions → physical GPU memory blocks. Same principle as a page table, one level down.

How Paged
Attention Works.

# Logical view (what the model sees)
seq = [t0, t1, t2, ... t255]

# Physical layout (GPU memory)
block_table = {
  0: block_7, # t0–t15
  1: block_2, # t16–t31
  2: block_14, # t32–t47
} # non-contiguous, zero waste

Copy-on-write enables free beam search. Multiple beams share physical blocks until they diverge, with no duplication until necessary.

Quick
Reference.

KV Cache
Stores K/V pairs from attention layers. Avoids O(n²) recomputation. Grows with sequence length.
Frag.
Contiguous pre-allocation wastes 60–80% of GPU memory reserving for worst-case sequence lengths.
PagedAttn
Non-contiguous KV blocks mapped via a block table. Eliminates fragmentation and internal waste.
Block table
Maps logical sequence positions → physical GPU blocks. Direct analogy to an OS page table.
vLLM
First production implementation. 2–4× throughput gains. Standard for high-throughput LLM serving.
