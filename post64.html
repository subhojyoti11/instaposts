The Castle
and Moat Is Gone.

Old model
Build a hard perimeter (firewall, VPN) and trust everything inside it. Employees on the corporate network could reach internal services without additional authentication.
The problem
The perimeter has dissolved. Employees work from home. Services run in the cloud. Partners need access to internal APIs. The inside of the network is no longer safe by definition.
Lateral move
When attackers breach the perimeter, they move freely. A compromised laptop on the internal network can reach HR databases, payment systems, and source code with no additional challenge.
The answer
Zero Trust assumes the network is already compromised. Every request must prove who it is, what device it is on, and what it is authorized to access, regardless of network location.

03 / 07

The Three
Core Principles.

Verify explicitly
Authenticate and authorize every request using all available signals. User identity, device health, location, service, and data classification all factor into each access decision, every time.
Least privilege
Grant the minimum permissions required for the specific task, for the minimum time. Just-in-time access replaces standing permissions. Developers get temporary elevated access, not permanent admin rights.
Assume breach
Design as if the attacker is already inside. Segment everything. Encrypt all traffic end-to-end, even on internal networks. Log and inspect all requests. Detect lateral movement before it reaches critical systems.

Location is no longer a trust signal. A request from inside the data center gets the same scrutiny as a request from a coffee shop.

The Policy Engine
in Practice.

# OPA policy: every request evaluated here package zerotrust.authz default allow = false allow { input.user.identity == verified input.device.posture == "compliant" input.device.encrypted == true valid_scope[input.resource] } valid_scope[r] { role_permissions[input.user.role][r] # checked on every single request }

OPA (Open Policy Agent) evaluates this policy on every API call. No request is implicitly trusted based on source.

05 / 07

The Five
Pillars.

Identity
Verify every human and machine identity using MFA, certificates, or workload identity. Identity is the new perimeter. OIDC and SPIFFE/SPIRE issue short-lived credentials to workloads.
Devices
Check device posture before granting access. Is the OS patched? Is disk encryption enabled? Is endpoint detection running? Non-compliant devices are quarantined, not trusted.
Network
Micro-segment the network so services can only reach services they need. Service mesh with mutual TLS (mTLS) encrypts and authenticates all service-to-service traffic on the internal network.
Workloads
Every container, function, and VM gets a cryptographic identity. SPIFFE IDs are issued at runtime. A service proves what it is with a certificate, not with a network address or IP.
Data
Classify data and enforce access by classification. PII and financial records require stronger identity verification and device posture checks than public content APIs.

06 / 07

In
Production.

Google BeyondCorp
Google deployed Zero Trust internally after the 2009 Operation Aurora breach. By 2014 every Google employee reached internal apps through a context-aware proxy with no VPN. BeyondCorp became the blueprint.
Istio
Service mesh for Kubernetes that enforces mTLS between every pod by default. No code changes required. The sidecar proxy handles certificate rotation and policy enforcement at the network layer.
Cloudflare Access
Replaces VPN for browser-based access to internal applications. Every request passes through Cloudflare's edge, evaluated against identity and device posture policies before reaching the origin.
HashiCorp Vault
Dynamic secrets with short TTLs replace static credentials. Database passwords and API keys are generated on demand, valid for minutes, and automatically revoked. No secret is stored in code or config files.

07 / 07

Quick
Reference.

Zero Trust
Security model that eliminates implicit trust based on network location. Every request is authenticated, authorized, and encrypted regardless of whether it originates inside or outside the corporate network.
mTLS
Mutual TLS: both sides of a connection present certificates. Istio enforces mTLS between Kubernetes pods automatically. Services authenticate to each other, not just to a load balancer.
SPIFFE
Secure Production Identity Framework for Everyone. Issues cryptographic workload identities at runtime. A container gets an SVID (certificate) it uses to prove identity to other services.
OPA
Open Policy Agent: a general-purpose policy engine. Evaluates authorization policies for API gateways, Kubernetes admission, and service meshes using Rego policy language.
JIT access
Just-in-time access: temporary elevated permissions granted on demand. Replaces standing admin rights. A developer gets 30-minute database access for a specific task, then credentials auto-expire.
