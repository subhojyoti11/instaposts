From 32 Bits
to 1.58.

float32
32 bits per weight. Training standard. Full precision. 4 bytes per parameter. A 7B model = 28 GB.
float16
16 bits. Inference standard. Half the memory. Same multiply hardware. A 7B model = 14 GB.
int8/4
8 or 4 bits via post-training quantization. Reduces memory 2–4x. Still uses integer multiplications. Quality degrades at int4.
1.58-bit
Ternary weights: {-1, 0, +1}. log2(3) = 1.58. No multiplications at inference. Hardware changes fundamentally.

Weights as
-1, 0, +1.

# Standard matmul: multiply-accumulate
output = x @ W # W: float16 or int8
# requires N*M multiply ops

# BitNet b1.58: ternary weights
W in {-1, 0, +1} # no fractions
output = sum(x[i] if W[i]==+1)
       - sum(x[i] if W[i]==-1)
# W[i]==0: skip, free computation

1.58 bits = log2(3). Three states: amplify (+1), suppress (-1), ignore (0). Matrix multiplication becomes addition and subtraction only. Zero multiplies at inference.

04 / 07

Why Hardware
Changes.

Multiply
Floating-point multiply requires ~28 transistor ops. Integer add requires ~3. Eliminating multiplications is a ~10x energy gain per operation.
Memory
1.58-bit weights cut memory ~10x vs float16. A 7B model fits in ~1.6 GB. Runs in a phone's RAM. No GPU required.
CPU-first
Ternary math maps to SIMD integer instructions. ARM NEON and x86 AVX run 1.58-bit inference faster than float16 on GPU at batch size 1.
Future
Purpose-built 1.58-bit hardware does not yet exist. When it does, today's GPU instruction sets become unnecessary for inference entirely.

Quick
Reference.

Quant
Quantization. Reducing weight precision. float32 → float16 → int8 → int4 → 1.58-bit. Each step trades memory for fidelity.
BitNet
Microsoft Research, 2023. Binary weights {-1, +1}. Matrix multiply becomes XOR + popcount. Foundation for b1.58.
b1.58
Ternary extension: {-1, 0, +1}. log2(3) = 1.58 bits. Better accuracy than binary. Zero multiplications at inference.
STE
Straight-Through Estimator. Passes gradient through quantization unchanged. Required for training, without it, ternary models cannot learn.
Use when
Edge, mobile, CPU-only, energy-constrained. Anywhere a 7B float16 model is too large or too expensive to run continuously.

Caption
A 7 billion parameter model that fits in 1.6 gigabytes. No GPU. No data center. Runs on your phone. Standard neural network weights are stored as 16-bit floating-point numbers. Each one is a fraction. Each one requires a multiplication to use. A 7B parameter model holds 7 billion of them. Running it means 7 billion floating-point multiplications per token, per layer. That is why serving large language models requires expensive GPU clusters and consumes enough electricity to power a small building. BitNet asks a different question. What if the weights were not fractions at all? Microsoft Research published BitNet in 2023. The idea: replace every weight in every linear layer with a binary value, either -1 or +1. Nothing else. No fractions. No floating-point arithmetic. A matrix multiplication becomes a sequence of additions and subtractions. On modern hardware, that is roughly 10 times cheaper per operation in both energy and transistor count. BitNet b1.58 extended this to ternary weights: -1, 0, or +1. Three states. Log2 of 3 is 1.58, hence the name. The zero state adds something binary could not offer, the ability to skip a weight entirely, contributing nothing to the output. Ternary weights match the accuracy of binary models while being easier to train, and at 3 billion parameters and above they match the quality of full float16 models on standard benchmarks. The training trick is subtle. The weights are stored in full float32 precision during training and updated normally by the optimizer. Only in the forward pass are they quantized to -1, 0, or +1 for the actual computation. Gradients flow back through the quantization step unchanged, this is called the straight-through estimator. Without it, the discrete step function would zero out every gradient and the model could never learn. The result is a model that is 8 to 10 times smaller, uses a tenth of the energy per token, and runs faster on a CPU than a float16 model runs on a GPU, for the single-request batch sizes that matter most for edge and mobile deployment. We have been building AI infrastructure around the assumption that weights are floats. BitNet suggests that assumption was never necessary.
