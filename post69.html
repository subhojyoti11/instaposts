The Cloud
Privacy Problem.

Today's model
You encrypt data before sending it to the cloud. The cloud decrypts it, runs the computation, returns the result. Your data is exposed in plaintext inside the server during every operation.
The risk
The cloud provider, a rogue employee, or an attacker with server access sees everything. Healthcare records, financial models, genomics data — all exposed the moment computation begins.
The ideal
Send encrypted data to the cloud, receive an encrypted result, decrypt locally. The cloud performs useful work on data it can never read. This is exactly what homomorphic encryption provides.
Why now
In 2009, Craig Gentry published the first fully homomorphic encryption scheme. Performance has improved by over a million times since then. Production deployments are real today.

03 / 07

How HE
Actually Works.

The core idea
HE schemes define addition and multiplication operations that work directly on ciphertexts. The result of operating on encrypted values, when decrypted, equals the result of operating on the plaintexts.
Partial HE
Supports only one operation, either addition or multiplication, but not both. RSA is multiplicatively homomorphic. Paillier cryptosystem is additively homomorphic. Used in secure voting and e-cash.
Somewhat HE
Supports both addition and multiplication but only for a limited depth of operations. Each operation introduces noise. After enough multiplications, the noise overwhelms the signal and decryption fails.
Fully HE
Bootstrapping resets the noise level, allowing unlimited operations. Gentry's breakthrough was showing that the decryption circuit itself can be evaluated homomorphically to refresh a ciphertext and continue computing.

The Schemes
Compared.

BFV / BGV
Optimized for integer arithmetic. Used for exact computations on discrete values: counting, database queries, set membership tests. Preferred for privacy-preserving database lookups.
CKKS
Optimized for real and complex numbers with approximate arithmetic. Ideal for machine learning inference on encrypted data. Google's HEIR compiler targets CKKS for neural network evaluation in the cloud.
TFHE
Supports arbitrary boolean circuits with very fast bootstrapping. Each gate takes about 13ms. Used by Zama.ai for encrypted database queries and confidential smart contracts on Ethereum.
FHEW
Fast gate-by-gate bootstrapping, lower latency than TFHE for small circuits. Best for binary decision problems: fraud detection, credit scoring, KYC checks where the yes/no answer stays encrypted.

06 / 07

In
Production.

Microsoft SEAL
Open-source HE library supporting BFV, BGV, and CKKS. Used inside Azure for confidential cloud computing. Allows customers to run ML inference on sensitive data without exposing it to Microsoft.
Google HEIR
High-level Intermediate Representation for HE, built on MLIR. Compiles standard ML models into CKKS operations automatically. Eliminates manual HE programming for ML inference pipelines.
Zama.ai
Builds Concrete and TFHE-rs, production FHE compilers. Partners with Inco Network for confidential Ethereum smart contracts. Input data, computation logic, and outputs stay encrypted end-to-end.
IBM HElayers
C++ toolkit for AI inference under CKKS. IBM demonstrated logistic regression and neural network inference on encrypted genomic data. Healthcare is the primary target for confidential genomics analysis.

07 / 07

Quick
Reference.

HE
Homomorphic Encryption: perform computations on encrypted data without decrypting. The result, when decrypted, equals what you would get from computing on the plaintexts directly.
FHE
Fully Homomorphic Encryption: supports unlimited additions and multiplications via bootstrapping. Craig Gentry's 2009 breakthrough. Bootstrapping resets ciphertext noise to allow continued computation.
CKKS
Approximate HE scheme for real-number arithmetic. Ideal for ML inference. Google HEIR and Microsoft SEAL both implement CKKS for neural network evaluation on encrypted inputs.
TFHE
Fast bootstrapping HE for boolean circuits. Each gate operation takes roughly 13ms. Used by Zama.ai for encrypted database queries and confidential smart contracts without revealing inputs.
Noise
Each HE operation adds noise to the ciphertext. Too much noise makes decryption fail. Bootstrapping resets the noise level at the cost of significant computation overhead per refresh.

Caption
Every time your data reaches the cloud, it gets decrypted. That is the moment it is most vulnerable. The standard model of cloud computing has an uncomfortable truth at its center. You encrypt data before transmission, but the cloud provider must decrypt it before running any computation. During that window, your data is plaintext inside a server you do not control. Healthcare records, financial models, genomic sequences — all exposed to anyone with access to the server, whether a rogue employee, a subpoena, or a sophisticated attacker. Homomorphic encryption eliminates that window entirely. An HE scheme defines addition and multiplication operations that work directly on ciphertexts. When you decrypt the result, you get the same answer you would have gotten by operating on the original data. The cloud performs useful computation on data it can never read. Craig Gentry published the first fully homomorphic encryption scheme in 2009 as part of his Stanford PhD thesis. It was theoretically groundbreaking and computationally impractical. Performance has improved by over a million times since then. Production deployments are real today. The CKKS scheme handles real-number arithmetic with approximate results, making it ideal for machine learning inference. Google's HEIR compiler takes standard neural network models and compiles them into CKKS operations automatically. Microsoft SEAL implements CKKS inside Azure for confidential cloud computing. TFHE supports arbitrary boolean circuits and enables Zama.ai to run encrypted database queries and confidential Ethereum smart contracts where inputs, logic, and outputs stay encrypted end-to-end. The remaining constraint is speed. Bootstrapping, the operation that resets ciphertext noise to allow continued computation, is expensive. Current FHE is 100x to 10,000x slower than plaintext computation depending on the scheme. For narrow, high-value applications like genomics analysis and financial risk scoring, that overhead is worth paying.
