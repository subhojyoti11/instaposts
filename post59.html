02 / 07

The
Collaboration Problem.

Problem
Two users edit the same document simultaneously, offline or on slow connections. When they reconnect, whose version wins?
Naive
Last-write-wins discards one user's work entirely. Manual conflict resolution breaks collaborative flow. Neither is acceptable at production scale.
OT
Operational Transform, the approach Google Docs uses. Transforms operations relative to concurrent operations. Works, but requires a central server to serialize all updates. Remove the server and OT breaks.
Goal
A data structure that any node can update independently and that always converges to the same state, with no central coordinator, no conflict errors, ever.

03 / 07

The Three
Guarantees.

Commut.
merge(A, B) = merge(B, A). Order of receiving updates does not matter. Two nodes that see the same updates in different order produce identical state.
Assoc.
merge(A, merge(B, C)) = merge(merge(A, B), C). Batching updates doesn't matter. Intermediate states can be merged in any grouping.
Idemp.
merge(A, A) = A. Receiving the same update twice produces the same result as receiving it once. Duplicate delivery is safe and expected.
Result
Any two nodes that have seen the same set of updates will have identical state, regardless of order received, number of times received, or network partition duration.

Counter.
Register. Set.

# G-Counter (grow-only, per-node):
state = { A: 3, B: 5, C: 1 }
value() → sum(state.values()) # = 9
merge(x,y) → { k: max(x[k],y[k]) }

# OR-Set (add wins on concurrent remove):
add(item) → set.add( (item, unique_tag) )
remove(item) → remove all tags of item
has(item) → any tag for item present

# LWW-Register (last write wins):
merge(a,b) → a if a.ts > b.ts else b

05 / 07

Sequence CRDTs
for Text.

RGA
Replicated Growable Array. Each character gets a unique logical timestamp. Insertions ordered by timestamp with deterministic tie-breaking. Used in ShareDB.
YATA
Yet Another Transformation Approach. Used in Yjs, one of the most widely deployed CRDT libraries. Powers ProseMirror, Quill, and CodeMirror collaborative integrations.
Automerge
JSON CRDT built on RGA. Supports nested maps, lists, text, counters. Handles the full document model, not just plain text. Used by Ink & Switch.
Tree CRDT
Notion's block model uses a tree CRDT. Each block has a CRDT-based parent pointer. Moving a block while another user deletes its parent requires special cycle-prevention logic.

Figma
Multiplayer design canvas runs on CRDTs. Every property change, node move, and style update is a CRDT operation. Concurrent edits from thousands of users, zero conflicts.
Notion
Block-based CRDT model. Each block has a unique ID and a CRDT-based parent pointer. Offline editing merges cleanly on reconnect, no user prompt, no data loss.
Redis
CRDT module for geo-distributed clusters. Counters, sets, and registers replicate across data centers without coordination overhead or distributed locks.
CloudKit
Apple's iCloud sync uses CRDT-like semantics. LWW with vector clocks for complex document merges. Every iPhone note and reminder syncs without a conflict dialog.

07 / 07

Quick
Reference.

CRDT
Conflict-free Replicated Data Type. Merges states from multiple nodes with mathematical guarantee of eventual consistency. No central coordinator required.
3 Props
Commutativity, associativity, idempotency. Together they make merge order-independent and safe to apply any number of times with any batching.
G-Counter
Grow-only counter. Each node tracks its own count; merge takes max per node. The simplest CRDT, no coordination, no conflicts, mathematically proven.
OR-Set
Observed-remove set. Concurrent add and remove of the same element: add wins. Avoids the lost-update problem without timestamps or consensus.
RGA/YATA
Sequence CRDTs for text. Unique logical timestamps per character. Enables conflict-free collaborative editing without a central operation serializer.

Caption
Two users edited the same document at the same time. Both were offline. Both were right. When they reconnected, one of them lost their work. That is how most conflict resolution works. Last write wins. One clock is trusted. One version survives. It is simple and wrong. Real-time collaboration requires a different model entirely. Operational Transform was the first serious answer. Google Docs uses it. The idea: transform every operation relative to every concurrent operation before applying it. If Alice deletes character 5 while Bob inserts at position 5, transform Bob's insert to account for Alice's delete. It works, but it requires a central server to serialize all operations into a canonical order. Remove the central server and OT falls apart. The transformation functions stop being correct when operations arrive in arbitrary order from arbitrary nodes. CRDTs take a different approach. Instead of transforming operations, design the data structure so that any two states can always be merged into the same result, regardless of the order you merge them. Three mathematical properties make this possible: commutativity (merge A then B equals merge B then A), associativity (grouping doesn't matter), and idempotency (merging the same state twice equals merging it once). A data structure with these three properties can replicate to any number of nodes, accept updates in any order, and always converge to identical state. The simplest CRDT is the G-Counter: each node tracks its own count, and merge takes the max per node. The OR-Set solves the harder problem: concurrent add and remove of the same element. Add wins, always. The Replicated Growable Array makes this work for text: each character gets a unique logical timestamp, and insertions are ordered by timestamp with deterministic tie-breaking. Figma's multiplayer canvas runs on CRDTs. Notion's block model uses CRDTs. Apple's CloudKit uses CRDT-like semantics for iCloud sync. Redis has a CRDT module for geo-distributed clusters. The insight is not new. The formalization dates to 2011. But it matters now because collaboration is distributed, connections are unreliable, and users work offline. You cannot afford to ask a server what the right answer is.
