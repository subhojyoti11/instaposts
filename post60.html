02 / 07

The PCIe
Bottleneck.

Discrete
Traditional GPU systems: CPU has its own DRAM, GPU has its own VRAM. Connected by PCIe. Separate address spaces, separate physical memory, separate buses.
Bandwidth
PCIe 4.0 x16: ~32 GB/s. Your GPU's internal HBM bandwidth: 900+ GB/s. The bus is 28x slower than the memory it feeds. Every cudaMemcpy pays this cost.
Explicit
Every ML operation that needs GPU compute requires an explicit copy: cudaMalloc, cudaMemcpy, then compute, then cudaMemcpy back. The programmer owns this overhead.
CUDA UM
NVIDIA's Unified Memory (cudaMallocManaged) makes copies transparent, but not free. Pages still migrate across PCIe on demand. The bus tax is real; it is just hidden.

03 / 07

One Die.
One Pool.
No Bus.

On-die
Apple's UMA puts CPU, GPU, Neural Engine, and ISP on the same die. All connected to a single shared LPDDR5/X memory pool. No PCIe. No discrete VRAM.
Bandwidth
M4 Max: 546 GB/s shared across all processors. Every compute unit on the chip sees the same memory at the same bandwidth. No partitioning.
Pointer
The CPU writes a tensor. The GPU reads it from the same physical address. No copy. No migration. Pass a pointer, not a matrix. The OS never moves the data.
Trade-off
Memory cannot be upgraded after purchase. Every GB the OS uses is a GB the GPU does not have. Capacity is shared, not dedicated to any one processor.

Pass a Pointer,
Not a Matrix.

# Discrete GPU (3 copies per forward pass):
cpu_t = torch.tensor(data) # CPU RAM
gpu_t = cpu_t.to("cuda") # PCIe copy: ~32 GB/s
out = model(gpu_t) # GPU computes
result = out.cpu() # PCIe copy back

# Apple Silicon UMA (zero copies):
import mlx.core as mx
tensor = mx.array(data) # unified memory
result = model(tensor) # CPU+GPU share address
# no copy at any step
# 546 GB/s to all cores

05 / 07

The
Numbers.

PCIe 4.0
32 GB/s peak bidirectional bandwidth between CPU and discrete GPU memory. The wall every cudaMemcpy hits.
M4 Max
546 GB/s shared memory bandwidth. Available to CPU, GPU, and Neural Engine simultaneously. 17x faster than the PCIe bus it replaces.
70B model
A 70B parameter model at 4-bit quantization (35 GB) fits entirely in M2 Ultra's 192 GB. The full context is always in fast memory. No GPU memory splitting needed.
H100
80 GB HBM3 at 3.35 TB/s, raw bandwidth still higher than M4 Max. For large batch training workloads, discrete GPUs maintain the advantage. UMA wins for single-machine inference.

In

MLX
Apple's ML framework designed from the ground up for UMA. Operations spanning CPU and GPU are zero-copy by design. No memcpy in the critical path, no boundary to cross.
llama.cpp
Runs faster on Apple Silicon than expected because large quantized models fit entirely in unified memory. No layer offloading, no CPU-GPU splits, no PCIe stalls mid-inference.
Ollama

Context
Large context windows benefit most. A 200K-token KV cache that would overflow a discrete GPU's VRAM fits comfortably in M2 Ultra's 192 GB unified pool.

Quick
Reference.

UMA
Unified Memory Architecture. CPU, GPU, and Neural Engine on one die, sharing a single LPDDR5/X pool. No PCIe bus. No discrete VRAM. One address space.
PCIe cost
32 GB/s peak on PCIe 4.0 x16. Every cudaMemcpy on a discrete GPU pays this bandwidth tax. UMA eliminates the bus and the copy it necessitates.
Bandwidth
M4 Max: 546 GB/s shared. All processors access the same physical memory at the same speed. 17x the PCIe bandwidth it replaces for cross-processor data sharing.
MLX
Apple's ML framework built for UMA. Zero-copy between CPU and GPU by design. The fastest way to run inference and fine-tuning on Apple Silicon.
Limit
Memory is fixed at purchase and shared with the OS. H100 still leads on raw bandwidth and capacity for large-batch training. UMA wins on single-machine inference at scale.

Caption
Every time you call cudaMemcpy, you are waiting. Waiting for data to travel from CPU memory across the PCIe bus to GPU memory. At PCIe 4.0 speeds, that bus runs at about 32 gigabytes per second. Your GPU's internal memory bandwidth is 900 gigabytes per second. The bottleneck is not the GPU. It is the bus you use to feed it. Discrete GPU systems are built around a fundamental separation: the CPU has its own DRAM, the GPU has its own VRAM, and they are connected by PCIe. Every time the GPU needs data the CPU prepared, you copy it. Every time the CPU needs GPU results, you copy it back. NVIDIA's Unified Memory (cudaMallocManaged) makes this transparent but does not eliminate it. Pages still migrate across PCIe on demand. The bus tax is still real. It is just hidden. Apple's Unified Memory Architecture removes the bus. The CPU, GPU, Neural Engine, and ISP all live on the same die, connected to a single shared memory pool. There is no discrete VRAM. There is no PCIe link between processor and memory. On an M4 Max, every processor on the chip shares 546 gigabytes per second of memory bandwidth. The CPU writes a tensor. The GPU reads it from the same physical address, no copy, no migration, no latency. A pointer is enough. This is why llama.cpp runs faster on Apple Silicon than expected. It is why a 70 billion parameter model quantized to 4-bit fits in the 128 GB M2 Ultra and runs well, while a discrete GPU system with 80 GB of VRAM struggles with the same model. The memory is not partitioned. The full capacity is available to every processor at once. MLX, Apple's machine learning framework, was designed from the ground up for UMA. Operations that span CPU and GPU are zero-copy by design. There is no memcpy in the critical path because there is no boundary to cross. Ollama uses the same principle to serve 70B models on Mac Studio with no layer offloading and no PCIe stalls mid-inference. The trade-off is real: memory cannot be upgraded after purchase, and the H100 still leads on raw bandwidth for large-batch training at 3.35 TB/s. But for inference, fine-tuning, and running large context windows on a single machine, UMA changed the constraint entirely. We didn't make the GPU faster. We removed the wall between the GPU and everything else.
