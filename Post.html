02 / 10● CRDT 101

What is a CRDT?

Conflict-free Replicated Data Type.

A CRDT is a data structure designed so that multiple copies of it, edited independently on different devices, can always be merged back together automatically, without a central server deciding who "wins."

The magic: the merge function is mathematical. No matter the order edits arrive in, everyone ends up with the exact same final state. Order-independent. Conflict-proof.

03 / 10● EXAMPLE

Alice & Bob write "HELLO"

Both are offline. Both edit the same shared document.

alice: inserts "HEL" at pos 0
bob:   inserts "LO" at pos 0
→ they reconnect
→ CRDT merges by unique IDs + timestamps
result: "HELLO" ✓
(both devices now identical)

No server asked. No "resolve conflict" dialog. Each character has a unique ID, and the merge rules guarantee everyone converges to the same string.

04 / 10● CRDT TYPES

The family tree

CRDTs split into two big branches,then fan out into specialized structures:

01
State-based (CvRDT)Nodes send their full state. Merge = mathematical join. Simple, heavy on bandwidth.
02
Operation-based (CmRDT)Nodes send only the operations. Lighter, but needs reliable delivery.
03
G-Counter / PN-CounterGrow-only and increment/decrement counters. For likes, views, votes.
04
G-Set / 2P-Set / OR-SetSets you can add to (and sometimes remove from) across replicas.
05
LWW-Register & Sequence CRDTsLast-Writer-Wins values, and RGA/Logoot for collaborative text.

06 / 10● HASHCHAIN 101

Now, what's a HashChain?

A tamper-evident ledger built from cryptographic breadcrumbs.

A HashChain is a sequence where each entry contains the cryptographic hash of the entry before it. Change any past record, and every hash downstream breaks; instantly visible to everyone.

Where CRDTs make edits mergeable, HashChains make history unforgeable. Different problem, opposite design philosophy.

08 / 10● HASHCHAIN TYPES

The shapes it takes

01
Linear Hash ChainThe classic: one block points to one previous block. Simple, sequential, append-only.
02
Merkle Tree (Hash Tree)Hashes arranged as a binary tree, verify huge datasets by checking just one root hash.
03
Merkle DAGDirected acyclic graph of hashes. Powers IPFS, Git, and content-addressed storage.
04
BlockchainHash chain + consensus algorithm + incentives. Bitcoin, Ethereum, etc.
05
Hash Time-Lock / One-Time ChainsLamport-style chains used for authentication, OTPs, and micropayment channels.

10 / 10● TAKEAWAY

So, which do you need?

They solve opposite problems. Pick by the question you're asking:

Reach for a CRDT when…

  • Multiple users edit the same thing
  • Offline-first matters
  • You need automatic merge
  • Eventual consistency is fine
  • Think: collaboration

Reach for a HashChain when…

  • History must be tamper-proof
  • You need verifiable order
  • Zero-trust environments
  • Audit trails & integrity
  • Think: provenance

CRDTs answer "how do we agree?" HashChains answer "how do we remember?" The best modern systems use both.
