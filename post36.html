Your code is lying to you.

You see a flat sequence of characters. The compiler sees a hierarchy of meaning.

What you write:
let score = 100;

What the compiler actually sees:
VariableDeclaration
│
├─ kind: "let"
│
└─ declarator
├─ id: Identifier "score"
└─ init: Literal 100

"Abstract" means the tree strips syntax noise. Semicolons, brackets, whitespace, all gone. let x=42; and let x = 42 ; produce the exact same tree. Only meaning survives.

The semicolons, the indentation, the formatting, all of it is decoration for humans. The compiler throws it away in stage one. It was never part of your program.

Three Stages.
Text → tokens → tree.

Why three stages? Separation of concerns. The lexer only knows characters. The parser only knows grammar. The AST only stores meaning. Each stage does one job perfectly.

Depth = Priority
3 + 4 * 5 - 1

Deeper = evaluates first. × sits at depth 3 → runs first (4×5=20). Then + at depth 2 (3+20=23). Then − at depth 0 (23−1=22). The tree IS the order of operations.

Complex Structure
function add(a, b) { return a + b }

One line of code. Seven tree nodes. Name, parameters, body block, return statement, binary expression, two identifier references. This is why compilers are complex; even simple code creates deep trees.

You use ASTs every single day.

ESLint
Walks every node in your AST. "no-unused-vars" finds Identifier nodes declared but never referenced. Your linter IS an AST visitor
Prettier
Parses → AST → reprints from scratch. It doesn't fix your formatting. It throws it away entirely and regenerates from the tree
Babel
Swaps AST nodes. ArrowFunction → FunctionExpression. Optional chaining → ternary. It rewrites your tree, then prints new code from it
TypeScript
Type-checks the tree. Every red squiggly = an AST node whose type annotation doesn't match its usage elsewhere in the tree
Webpack
Reads import/require from the AST to build the dependency graph. Knows which files need which, without executing anything

Code that rewrites code.

Babel doesn't use find-and-replace. It modifies the tree, then generates new code from it:

INPUT (ES6):
const add = (a, b) => a + b;

───── AST Transform ─────
Find: ArrowFunctionExpression
Replace: FunctionExpression
Add: ReturnStatement wrapper

OUTPUT (ES5):
var add = function(a, b) {
return a + b;
};

Facebook used AST codemods to migrate millions of lines from React.createClass to ES6 classes. Not regex. Structural, safe, syntax-aware rewrites. One script, entire codebase, zero manual edits.
