Reason, Act, Observe.

User: "Book a 30min meeting with Sarah tomorrow"
────────────────────────────
THINK: I need to check Sarah's calendar
and find a free 30min slot tomorrow
ACT: call calendar.getFreeSlots(sarah, tomorrow)
OBS: [10:00-10:30, 14:00-14:30, 16:00-16:30]
────────────────────────────
THINK: 3 slots available. Pick earliest.
ACT: call calendar.createEvent(sarah, 10:00)
OBS: ✓ event created, invite sent
────────────────────────────
RESPOND: "Booked with Sarah at 10am"

ReAct = Reasoning + Acting. The LLM thinks about what to do, calls a tool, observes the result, then thinks again. This loop repeats until the task is complete. The model decides WHEN to stop.

03 / 07ravenex.tech

What makes an agent.

LLM
The brain. Reads context, reasons about the task, decides next action. GPT-4o, Claude, Gemini. The model IS the decision engine
Tools
The hands. APIs the agent can call. Calendar, email, database, code execution, web search, file system. Each tool = a function definition
Memory
Short-term: conversation context. Long-term: vector DB storing past interactions, user preferences, learned facts. Persists between sessions
Planning
The strategy. Break complex tasks into subtasks. "Deploy to production" becomes: run tests → build → stage → verify → deploy → notify team

Agent = LLM + Tools + Memory + Planning. Remove any one component and you have a chatbot, not an agent. The LLM without tools can only talk. Tools without planning just execute blindly.

How tools actually work.

You register tools as function definitions:
────────────────────────────
tools: [
{name: "search_web",
params: {query: string}},
{name: "send_email",
params: {to, subject, body}},
{name: "run_sql",
params: {query: string}}
]
────────────────────────────
The LLM doesn't execute tools.
It outputs structured JSON saying WHICH tool to call with WHAT params.
Your code executes the call.
You return the result to the LLM.

The LLM is the router, not the executor. It decides which tool fits the user's request and formats the parameters. Your backend handles the actual API call, validation, and error handling. The model never touches your infrastructure directly.

The failure modes.

loop
Infinite loops. Agent calls tool, gets error, retries same tool, gets error, retries... Burns tokens endlessly. Always set a max_iterations cap
ghost
Hallucinated tool calls. Agent invents a tool that doesn't exist. Or passes wrong parameters. "run_sql" with natural language instead of SQL
scope
Scope creep. Asked to "clean up the code." Deletes files, refactors the architecture, changes the database schema. No boundaries defined
cost
Token explosion. A 15-step agent workflow with tool results at each step can consume 50K+ tokens. One complex task = $0.50-$2.00

Claude Code + terraform destroy. An AI agent ran a destructive command on a production database. 1.94M rows deleted. The agent didn't understand the consequences of the tool it was calling. This is the #1 risk of agentic AI in 2026.

07 / 07ravenex.tech

Rules for production.

01
Human-in-the-loop. For destructive actions (delete, deploy, send), require explicit human approval. Never auto-execute irreversible actions
02
Sandboxing. Agents run in isolated environments. No access to production databases, credentials, or infrastructure without explicit grants
03
Max iterations. Cap the ReAct loop at 10-15 steps. If the agent hasn't solved it by then, escalate to a human. Prevents infinite loops and token burn
04
Tool permissions. Read-only by default. Write access requires explicit scope. "Read email" ≠ "send email." Principle of least privilege
05
Audit logging. Log every tool call, every parameter, every result. When something goes wrong (and it will), you need the full trace

The 2026 truth: Agents are the most powerful and most dangerous AI pattern. They can 10× your team's output. They can also delete your production database. The difference is guardrails, not intelligence.
