The Relational Standard

B-Trees.
The Default Engine.

The B-Tree is an elegant, balanced search tree that keeps data sorted and allows for highly efficient reads, insertions, and deletions.

It updates data in-place on the disk. This makes reading incredibly fast because you know exactly where to look. It's the engine that powers the world's most trusted relational databases.

PostgreSQL MySQL / InnoDB SQLite

The catch? Writing random data forces the B-Tree to reorganize its pages, causing slow, random I/O on disk.

The Write Heavyweight

LSM Trees.
The Speed Demon.

Log-Structured Merge-Trees turn all writes into sequential appends. Instead of hunting for where to update data, they just write it to a memory buffer (MemTable), then flush it directly to disk (SSTable).

Because disks (even SSDs) love sequential writes, LSM Trees can absorb massive, relentless write volumes without breaking a sweat.

Cassandra RocksDB DynamoDB

The catch? To read data, the engine might have to scan multiple SSTables and merge them, making reads more expensive.

Physics of Storage

The Battle of
Amplification.

B-Tree
High Write Amplification. Updating a 10-byte row might force the database to rewrite a 4KB or 8KB page to disk.
LSM Tree
High Read Amplification. Reading a single row might require checking the MemTable, Bloom Filters, and 5 different SSTables.
Space
Space Amplification. LSM Trees leave obsolete data on disk until a background "compaction" process cleans it up. B-Trees overwrite obsolete data immediately.

The Verdict

Which Engine
Wins?

Don't overcomplicate it. The vast majority of applications are read-heavy.

Choose B-Trees (Postgres/MySQL) for 95% of your apps. They offer predictable performance, fast reads, and instant consistency. You won't outgrow a well-tuned Postgres database easily.

Choose LSM Trees (Cassandra/RocksDB) when you are building an analytics ingestion engine, a time-series database, or need to log millions of events per second with extreme write throughput.

Know the data structures, and you'll never choose the wrong database.
