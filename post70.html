What Vector
Databases Do.

Core idea
Store data as high-dimensional numeric vectors called embeddings. An embedding captures semantic meaning: similar documents, images, or concepts end up close together in the vector space regardless of exact wording.
The query
You embed your query and ask for the nearest neighbors. Instead of exact string matching, you search by meaning. "What are the side effects of aspirin?" finds articles discussing NSAIDs and gastropathy even without those exact words.
ANN search
Exact nearest-neighbor search is O(n) for n vectors. Approximate Nearest Neighbor (ANN) algorithms like HNSW and IVF-Flat reduce this to O(log n) by building a graph index over vectors, trading perfect accuracy for speed.
Use cases
RAG pipelines, semantic search, recommendation systems, image similarity. Any task where the question is: what is conceptually similar to this input? Pinecone, Weaviate, Qdrant, and pgvector handle this workload.

03 / 07

What Graph
Databases Do.

Core idea
Store data as nodes connected by typed, directional edges. The structure of the connections is the data. A user follows another user. A drug inhibits an enzyme. A transaction flows between two accounts.
The query
Traverse relationships, not vectors. Cypher (Neo4j) and Gremlin let you ask: who are all the people within 3 degrees of connection of this node, filtered by edge type? Impossible to express cleanly in SQL or a vector similarity search.
Strength
Joins between related entities are O(1) per hop because relationships are stored as direct pointers. A relational database runs an expensive JOIN operation. A graph database follows a pointer. At 6 hops, the difference is millions of times faster.
Use cases
Fraud detection, knowledge graphs, identity access management, supply chain tracing. Any task where the question is: how are these entities connected, and what does that structure reveal? Neo4j, Amazon Neptune, and TigerGraph handle this.

04 / 07

The
Queries
Compared.

-- Vector: semantic similarity (pgvector) SELECT title, content FROM docs ORDER BY embedding <-> embed('drug side effects') LIMIT 5; -- Graph: relationship traversal (Cypher) MATCH (u:User {id: 'alice'}) -[:FOLLOWS*1..3]-> (friend:User)-[:PURCHASED]->(p:Product) WHERE p.category = 'electronics' RETURN friend.name, p.title;

The difference: vector search finds what is semantically similar, graph traversal finds what is structurally connected. Neither replaces the other.

05 / 07

When to
Use Which.

Vector wins
When similarity is the question. RAG for LLMs: retrieve the 5 most relevant document chunks for a query. Product recommendations based on user behavior embeddings. Image deduplication at scale. Semantic document clustering.
Graph wins
When relationships and path structure are the answer. Fraud ring detection: find all accounts within 2 hops of a flagged transaction. Access control: does this user have permission through any role chain? Supply chain: trace a component across all sourcing relationships.
Both together
Enterprise knowledge graphs increasingly combine them. Microsoft GraphRAG uses a graph of entity relationships extracted by an LLM, then does vector search within each graph community. Meaning and structure both contribute to retrieval quality.
Neither for OLTP
Vector and graph databases are not replacements for transactional relational databases. Postgres handles your user table. pgvector adds a column for embeddings. You compose, not replace, as workloads grow more complex.

In
Production.

Pinecone
Managed vector database with serverless and pod-based tiers. Used by Notion, Salesforce, and Hubspot for semantic search and RAG. Handles billions of vectors with sub-10ms p99 latency at production scale.
Weaviate
Open-source vector database with native multi-modal support. Stores text, images, and audio with built-in embedding generation. Deployed by AstraZeneca for drug discovery knowledge retrieval.
Neo4j
The dominant graph database, used by 75% of Fortune 100 companies. NASA uses Neo4j to track the knowledge dependencies across all their research projects. Airbnb uses it for fraud detection across transactions.
Microsoft GraphRAG
Combines LLM-extracted knowledge graphs with vector search. An LLM reads documents to extract entities and relationships, builds a Neo4j graph, then retrieves context using both graph traversal and vector similarity for answers requiring multi-hop reasoning.

07 / 07

Quick
Reference.

Embedding
A dense numeric vector representing semantic meaning. Generated by an encoder model like text-embedding-3-large. Semantically similar inputs produce nearby vectors in high-dimensional space.
HNSW
Hierarchical Navigable Small World: the dominant ANN index algorithm. Builds a layered proximity graph over vectors. Enables O(log n) approximate nearest-neighbor queries instead of O(n) brute-force scan.
Cypher
Neo4j's graph query language. Uses ASCII-art syntax like (a)-[:KNOWS]->(b) to express relationship patterns. Read with MATCH, write with MERGE/CREATE, traverse multi-hop paths with *1..3 notation.
GraphRAG
Retrieval-Augmented Generation using a knowledge graph as the retrieval layer. LLM extracts entities and relationships from documents into a graph, then retrieves multi-hop context for queries requiring reasoning across many documents at once.
pgvector
PostgreSQL extension adding a vector column type and ANN index support. The pragmatic choice: add vector search to your existing Postgres instance without operating a separate vector database service.

Two engineers disagree on what database to use. One says vector, the other says graph. They are probably both right, for different parts of the same problem. Vector databases and graph databases solve fundamentally different questions. A vector database answers: what is semantically similar to this input? A graph database answers: how are these entities connected, and what does that structure tell us? Vector databases store embeddings, high-dimensional numeric representations of meaning generated by encoder models. When you query, you embed your search and retrieve the nearest neighbors in that space. Similar concepts cluster together regardless of exact wording. A question about medication side effects finds relevant content about drug interactions even without shared keywords. This makes vector databases the natural backbone for RAG pipelines, semantic search, and recommendation systems. Graph databases store nodes and typed, directional edges. The structure is the data. A user follows another user. A transaction flows between two accounts. A drug inhibits an enzyme. When you query, you traverse paths. Cypher asks: give me all accounts within three hops of this flagged node, filtered by relationship type. At six hops, a graph database using pointer traversal is millions of times faster than a relational JOIN operation. The mistake is treating them as competitors. Microsoft's GraphRAG uses an LLM to extract entities and relationships from documents into a graph, then retrieves context using both graph traversal and vector similarity. Meaning and structure together produce better answers than either alone. For teams starting with Postgres, pgvector adds vector search as a column type without a new service.
