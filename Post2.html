02 / 10● THE THEOREM

You can't have all three.

In 2000, Eric Brewer proposed: any distributed data store can only guarantee two of three properties at the same time.

Network partitions will happen. So the real question is: when they do, do you sacrifice consistency or availability?

// brewer's conjecture, 2000 swipe →
03 / 10● DEFINITIONS

What C, A, P mean

C
ConsistencyEvery read gets the most recent write. All nodes see the same data at the same time. No stale reads, ever.
A
AvailabilityEvery request gets a response, even if it's not the latest data. The system never refuses to answer.
P
Partition ToleranceThe system keeps running even when network links between nodes break. Messages get lost or delayed.

In any real distributed system, P is non-negotiable, networks always partition. So the choice is always C vs A.

04 / 10● THE ANALOGY

The pizza explanation

You run two pizza shops across town. They share the same menu and prices.

One day, the phone line between your two shops goes down. (Partition!)

A customer at Shop A wants to order. But Shop B just changed the price of a Margherita from ₹200 to ₹250. Shop A doesn't know yet.

Option 1 (CP): Shop A says "Sorry, I can't serve you until I confirm the price." Consistent, but unavailable.

Option 2 (AP): Shop A serves the old ₹200 price. Available, but inconsistent.

You can't serve the right price and always be open and survive phone outages. Pick two.

05 / 10● CP SYSTEMS

C + P, correct or nothing

Sacrifice availability. When partitioned, block requests until consistency is restored.

// network partition happens
node_A ←✕→ node_B
────────────────────────
client writes to node_A:
SET price = 250
────────────────────────
client reads from node_B:
ERROR: cannot guarantee consistency
→ request blocked until partition heals

You get: Every read is guaranteed accurate.
You lose: The system goes down during network issues.

// rather be right than fast swipe →
06 / 10● AP SYSTEMS

A + P, always answer

Sacrifice consistency. Always respond, even with potentially stale data.

// network partition happens
node_A ←✕→ node_B
────────────────────────
client writes to node_A:
SET price = 250 ✓ accepted
────────────────────────
client reads from node_B:
GET price → 200 ⚠ stale!
→ will sync when partition heals

You get: 100% uptime, always a response.
You lose: Different nodes may return different answers.

// better wrong than down swipe →
07 / 10● CA SYSTEMS

C + A, the unicorn

Sacrifice partition tolerance. Consistent and available, but only if the network never fails.

// no partition (everything is fine)
node_A ←--→ node_B ✓ connected
────────────────────────
write: SET price = 250 ✓ synced
read: GET price → 250 ✓ correct
────────────────────────
⚠ but if network partitions...
→ entire system goes down

In practice: This is a single-node database. PostgreSQL, MySQL traditional RDBMS. They don't distribute, so partitions aren't a concern. But they also don't scale horizontally.

08 / 10● IN THE WILD

Every database picked a side

Here's where the systems you use actually land:

MongoDBCP by default. Blocks reads during elections to stay consistent.
Redis ClusterCP. Rejects writes during partitions if quorum is lost.
CassandraAP. Tunable consistency but default is availability-first.
DynamoDBAP. Eventually consistent reads by default. Fast, always up.
PostgreSQLCA. Single-node, strongly consistent. No partition tolerance.
CouchDBAP. Multi-master replication. Conflict resolution on merge.

09 / 10● THE NUANCE

It's not binary

Modern systems don't just "pick two", they tune the dial.

Real systems offer a spectrum between C and A, adjustable per query:

// Cassandra: tune per query
SELECT * FROM orders
WITH CONSISTENCY ONE; ← fast, AP

SELECT * FROM payments
WITH CONSISTENCY QUORUM; ← safe, CP

SELECT * FROM audit_log
WITH CONSISTENCY ALL; ← strict CP

This is "tunable consistency", you pick the trade-off for each operation. Orders can be eventually consistent. Payments can't. Same database, different guarantees.

10 / 10● CHEATSHEET

The decision matrix

Match your system to the trade-off you can live with:

CP

  • Banking
  • Inventory
  • Auth systems
  • Leader election
  • MongoDB, HBase, Redis Cluster

AP

  • Social feeds
  • Shopping carts
  • DNS
  • CDNs, caches
  • Cassandra, DynamoDB, CouchDB

CA

  • Single-server apps
  • Local databases
  • Small-scale CRUD
  • Dev/staging
  • PostgreSQL, MySQL, SQLite

There is no right answer, only the right trade-off for your problem. Know what you're giving up.
