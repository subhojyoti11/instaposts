Regular DBs can't
think about meaning.

-- SQL can only match exact words
SELECT * FROM docs
WHERE text LIKE '%refund%'
-- Misses: "return", "money back", "reimbursement"
────────────────────────────
-- Vector DB matches by meaning
search(embed("I want my money back"))
→ refund_policy.md sim: 0.91
→ return_process.md sim: 0.88
→ billing_faq.md sim: 0.82

The core problem: AI works with meaning, not keywords. A vector database stores float arrays and finds the nearest neighbors, that's what bridges language and math.

① EMBED (at ingest time)
"refund policy" → [0.21, -0.84, 0.55, ...]
1,536 floats stored in vector DB
────────────────────────────
② INDEX
DB builds HNSW graph over all
vectors for fast approximate search
────────────────────────────
③ QUERY (at runtime)
"money back" → [0.19, -0.81, 0.58, ...]
ANN search → top-k nearest → docs

Key insight: The vector DB never sees your text, only float arrays. The semantic understanding lives in the embedding model. The DB just stores and retrieves those floats fast.

HNSW, the index
that makes it fast.

Brute force: compare query to ALL 100M vectors
→ 100M dot products. ~10 seconds. Unusable.
────────────────────────────
HNSW (Hierarchical Navigable Small World)
Layer 2 (top) ─── few nodes, long shortcuts
Layer 1 (mid) ─── more nodes, medium links
Layer 0 (base) ── all nodes, fine-grained
────────────────────────────
Search: enter at top, greedily descend
→ finds ~nearest in O(log n)
→ 100M vectors in ~5ms. Usable.

ANN = Approximate Nearest Neighbor. HNSW doesn't guarantee the absolute closest vector - it finds one very close, very fast. In practice: 95-99% recall. The speed trade-off is almost always worth it.

Three ways to
measure similarity.

cos
Cosine similarity. Measures the angle between vectors. Ignores magnitude, same meaning regardless of text length. Best for sentence embeddings. Range: -1 to 1
dot
Dot product. Cosine × magnitude. Fast when vectors are normalized (unit length). OpenAI embeddings are pre-normalized, use this for speed. Range: -∞ to ∞
L2
Euclidean distance. Straight-line distance in N-D space. Good for images and spatial data where magnitude matters. Lower = more similar

Default choice: Cosine for text. Dot product if embeddings are normalized (slightly faster). L2 only for vision or spatial use cases.

Pick the right
tool.

Chroma
Start here. Open-source, runs in-process, zero infra. Perfect for prototypes. pip install chromadb and go
pgvec
Already on Postgres? pgvector adds a vector column. No new infra. Scales cleanly to ~1M vectors before ANN quality degrades
Pine
Serverless scale. Pinecone manages sharding, replication, and HNSW tuning. $0 up to 2M vectors. Best for production RAG with zero ops overhead
Wvt
Multi-modal + hybrid. Weaviate handles text, images, and audio. Built-in BM25 + vector hybrid search. Good for complex schemas
Qdrant
Open-source, no lock-in. Rust core, fast, payload filtering. Self-host or managed cloud. Pinecone-like features, fully open

The practical
cheatsheet.

01
Chunk at 200-500 tokens. Too small = no context. Too large = diluted meaning. Overlap 10-20%
02
Start with text-embedding-3-small. $0.02/MTok, 1,536 dims. Good enough for 90% of use cases
03
Threshold: 0.75+ signal, 0.85+ strong. Below 0.75 is noise. Tune per domain
04
Re-embed when switching models. Model A ≠ Model B. Switching = reprocess everything
05
Hybrid search beats pure vector. BM25 + vector catches names and codes embeddings smooth away
