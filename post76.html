02 / 07

The Memory
Wall.

The problem
LLM inference requires all model weights to fit in GPU VRAM simultaneously. A 70B model at float16 needs 140GB of VRAM. The cheapest GPU cluster with enough VRAM costs over $200,000.
Why VRAM?
GPU VRAM bandwidth is the bottleneck for token generation, not compute. Generating one token requires reading every weight at least once. VRAM at 2-3 TB/s is 10x faster than RAM and 1000x faster than NVMe.
Quantization
The first solution: reduce weight precision from float16 to int4, shrinking 140GB to 35GB. GGUF and llama.cpp use 4-bit quantization extensively. 35GB still exceeds a single consumer GPU.
The insight
Transformers process layers sequentially. During inference, only the weights of the current layer are needed. Layers not currently executing serve no purpose sitting in VRAM. AirLLM exploits this: stream layers from storage just-in-time and evict them immediately after use.

03 / 07

Layer
Streaming
Explained.

Normal inference
Load all 80 layers of Llama 70B into VRAM at startup. Requires 140GB VRAM. All 80 layers stay resident between tokens.
AirLLM inference
Load layer 1 from NVMe to RAM to VRAM, run it, evict it from VRAM, load layer 2, repeat. Peak VRAM is one layer (~1.75GB for a 70B model). Each token takes 60-120 seconds.
The memory hierarchy
NVMe SSD: 7 GB/s read speed. RAM: 50-100 GB/s bandwidth. VRAM: 2,000 GB/s bandwidth. NVMe delivers layers to RAM asynchronously while the GPU computes the current layer. Prefetching hides disk I/O latency.
Compression
AirLLM applies 4-bit quantization per layer before storing to disk. A 70B model weighs 35GB quantized instead of 140GB. Load time per layer drops from 20ms to 5ms.

05 / 07

Speed vs
Feasibility.

The cost
Layer streaming is slow. On an RTX 4060, AirLLM generates ~1 token per 60-120 seconds. A 200-token response takes 3-4 hours.
Use cases
Overnight document summarization, legal contract review, and research analysis where latency is irrelevant but cost matters. A $400 consumer GPU processes a 100-document corpus overnight that would cost $20 via API.
Apple Silicon advantage
Apple M-series chips have unified memory architecture: CPU and GPU share the same memory pool. M2 Max with 96GB unified memory runs 70B models at native speed via MLX without layer streaming.
Related approaches
llama.cpp implements model offloading that splits layers between GPU and CPU RAM. A 70B model with 24GB VRAM offloads 50 layers to GPU and 30 to CPU, achieving 3-5 tokens/s.

06 / 07

The
Ecosystem.

llama.cpp
The most widely used local inference engine, written in C++. Created by Georgi Gerganov in 2023. GGUF quantization and GPU/CPU layer offloading. The backbone of Ollama and LM Studio.
Ollama
Docker-like runtime for local LLMs built on llama.cpp. One command downloads and runs any model. Manages weights, GPU allocation, and an OpenAI-compatible API.
MLXApple's array framework optimized for Apple Silicon's unified memory. M2 Ultra with 192GB runs Llama 70B at 15 tokens/s, competitive with A100 at 1/10 the cost.
AirLLM
Open-source Python library for layer-by-layer inference on constrained hardware. Supports Llama 2, Llama 3, Mistral, and Falcon. Enables 70B+ on 8-24GB consumer GPUs at very slow speeds.

07 / 07

Quick
Reference.

AirLLM
Layer-by-layer inference: loads one transformer layer at a time from NVMe to VRAM, runs it, evicts it. Peak VRAM usage equals one layer size. Enables 70B models on 8GB GPUs at 1 token per 60-120 seconds.
Quantization
Reducing model weight precision from float16 to int4 or int8. Shrinks a 70B model from 140GB to 35GB (int4) with mild quality loss. GGUF is the standard format. llama.cpp, Ollama, and AirLLM all use quantization by default.
Layer offloading
Splitting model layers between GPU VRAM and CPU RAM. llama.cpp sets n_gpu_layers to control the split. Enables near-real-time inference on consumer hardware for models that overflow VRAM.
MLX
Apple's unified memory framework for Apple Silicon. CPU and GPU share the same physical memory pool. M2 Ultra 192GB runs 70B models at 15 tokens/s at 1/10 A100 cost.
GGUF
GPT-Generated Unified Format: the standard file format for quantized LLM weights. Replaced GGML. Supported by llama.cpp, Ollama, LM Studio, and koboldcpp.

Caption
A 70 billion parameter model requires 140GB of GPU VRAM to run. A consumer GPU has 8-24GB. AirLLM closes that gap by breaking the fundamental assumption that all weights must be resident simultaneously. Transformer inference is sequential by layer. During the forward pass, the GPU reads layer 1's weights, computes, then moves to layer 2. Layer 1's weights serve no purpose while layer 2 is computing. Normal inference keeps all layers in VRAM simultaneously for speed. AirLLM exploits the sequential structure to stream layers just-in-time: load layer 1 from NVMe to RAM to VRAM, run it, evict it, load layer 2, repeat. Peak VRAM usage drops from 140GB to the size of one layer, roughly 1.75GB for a 70B model. The cost is speed. An RTX 4060 running a 70B model via AirLLM generates approximately 1 token per 60-120 seconds. A 200-token response takes 3-4 hours. This is not interactive chat. It is a tool for offline batch processing where running a frontier-scale model matters but latency does not: overnight legal contract review, document summarization, and research analysis. After roughly 20 batch runs, the $400 consumer GPU has paid for itself versus API costs. AirLLM applies 4-bit quantization per layer before storing to disk, shrinking the 70B model from 140GB to 35GB. Combined with NVMe prefetching, load time per layer drops significantly, directly reducing time per token. The ecosystem offers alternatives for different constraints. llama.cpp implements layer offloading between GPU VRAM and CPU RAM, achieving 3-5 tokens per second on a 70B model with mixed GPU/CPU execution. Apple Silicon's unified memory architecture eliminates the problem entirely for M2 Ultra and M3 Ultra users. Apple's MLX framework runs 70B models at 15 tokens per second on 192GB unified memory.
