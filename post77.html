02 / 07

Why Static
Batching
Failed.

The old way
Early LLM serving systems used static batching: group N requests together, run them as a fixed batch, return all results together when the longest request finishes. Simple to implement, but catastrophically wasteful on GPU.
The waste
If one request in a batch of 32 generates 500 tokens while the rest generate 50, all 31 short requests sit idle waiting for the one long one to finish. GPU utilization collapses to near zero for 90% of the batch's wall-clock time. Throughput per dollar is terrible.
The insight
LLM inference is autoregressive: each token is generated one step at a time. Between each step, the GPU is idle for all completed sequences. Continuous batching replaces finished sequences with new requests immediately after they complete, keeping the GPU full at every step.
The result
Continuous batching improves throughput by 5-23x over static batching at equivalent latency. The UC Berkeley vLLM team published this in 2023 and demonstrated the improvement on real OpenAI API traffic patterns. It is now the standard approach in every production LLM serving system.

03 / 07

PagedAttention
Explained.

The KV cache
Attention mechanisms cache Key and Value tensors for all previous tokens to avoid recomputing them. For a 2,048-token context on a 70B model, the KV cache for a single request consumes several GB. Multiply by concurrent users and VRAM fills instantly.
The fragmentation
Before vLLM, KV caches were allocated contiguously with maximum sequence length reserved upfront. A request expected to generate 2,048 tokens reserved 2,048-token VRAM even if it generated only 20. Over 60-80% of reserved VRAM sat empty. GPU utilization remained low even with continuous batching.
PagedAttention
Borrowed from OS virtual memory: divide VRAM into fixed blocks and map non-contiguous blocks to logical KV cache pages. A request grows its KV cache one block at a time as tokens are generated. No upfront reservation. Fragmentation drops to under 4%. More requests fit in memory simultaneously.
Copy-on-write
PagedAttention enables KV cache sharing between requests with common prefixes. System prompt blocks are shared in memory across all concurrent users. When a request diverges, its page is forked via copy-on-write. Reduces VRAM usage by 30-50% for chat applications with long system prompts.

vLLM vs
TGI.

vLLM strengths
Highest throughput in independent benchmarks, widest model support, easiest Python integration. PagedAttention is the core innovation. Supports 100+ model architectures including Llama, Mistral, Qwen, and Gemma. OpenAI-compatible API out of the box. Favored for research and flexible production deployment.
TGI strengths
HuggingFace's production serving engine, battle-tested at scale by the HuggingFace Inference API. Better tooling for token streaming, request cancellation, and safety filtering. Supports FlashAttention and speculative decoding. Preferred by enterprise teams already on the HuggingFace ecosystem.
FlashAttention
Both vLLM and TGI use FlashAttention as the underlying attention kernel. FlashAttention rewrites attention computation to be IO-aware: it fuses operations to minimize GPU memory reads and writes. Results in 2-4x speedup on the attention computation itself, independent of batching strategy.
Speculative decoding
Both support speculative decoding: a small draft model generates candidate tokens, and the large model verifies them in parallel. Correct draft tokens are accepted without extra computation. Typical acceptance rates of 70-80% yield 2-3x throughput improvement on long-form generation tasks.

In
Production.

Anyscale
Deployed vLLM as the serving backbone for their LLM endpoints API. The vLLM team spun out from UC Berkeley RISELab to commercialize around the technology. Anyscale's endpoints serve hundreds of millions of tokens per day using vLLM on Ray Serve for distributed inference.
HuggingFace
TGI powers HuggingFace's Inference API, which serves models used by millions of developers daily. Battle-tested on peaks exceeding 1 billion requests per day across shared and dedicated inference endpoints. Reliability and monitoring tooling are mature compared to vLLM's younger ecosystem.
AWS Bedrock
Amazon uses continuous batching internally for all Bedrock model serving. Claude on Bedrock, Llama on Bedrock, and Titan models all use continuous batching to maximize GPU utilization. The approach is not publicly disclosed but confirmed in AWS re:Invent 2023 architecture sessions.
Modal
Serverless GPU compute platform that uses vLLM for its LLM serving primitives. Users deploy a vLLM inference endpoint with 5 lines of Python. Modal handles container orchestration, GPU allocation, autoscaling, and cold start optimization transparently. The underlying engine is vLLM with PagedAttention.

07 / 07

Quick
Reference.

Continuous batching
Replace finished requests with new ones at each generation step instead of waiting for the whole batch to finish. Improves throughput by 5-23x over static batching. Standard in all modern LLM serving systems since vLLM's 2023 paper.
PagedAttention
Manage KV cache in fixed-size blocks mapped non-contiguously, like OS virtual memory. Eliminates reservation waste. Enables KV cache sharing between requests with common prefixes via copy-on-write. Reduces VRAM fragmentation from 60%+ to under 4%.
FlashAttention
IO-aware attention kernel that fuses operations to minimize GPU memory reads. 2-4x faster than naive attention implementation. Used by both vLLM and TGI as the underlying compute kernel. Essential for long-context inference where attention is the bottleneck.
Tensor parallelism
Split model weight matrices across multiple GPUs, each computing a shard. An A100 80GB holds roughly a 70B model. 4x A100 via tensor parallel can serve a 70B model at 4x the throughput. Both vLLM and TGI support tensor parallel with the --tensor-parallel-size flag.
KV cache
Cached Key-Value tensors from previous attention computations to avoid recomputing them. Essential for autoregressive generation. Size scales linearly with sequence length and model size. PagedAttention's primary contribution is managing KV cache allocation efficiently.
