The Sequential
Trap.

1 token
One full forward pass. Every parameter, every step. No shortcuts, no batching.
Mem BW
Not compute, memory bandwidth is the real bottleneck. Weights must stream from HBM each step.
~97%
GPU compute sits idle waiting for the next memory round-trip to complete.

Generating 100 tokens = 100 sequential passes. The model isn't slow. The memory access pattern is.

03 / 07

Draft First.
Verify Later.

Draft Model | small & fast
Generates K candidate tokens autoregressively. Cheap. Approximate. Good enough.

Target Model | big & accurate
Runs one forward pass over all K candidates simultaneously. In parallel.

The large model verifies K tokens in the same wall-clock time it takes to generate one. That's the entire trick.

04 / 07

The Algorithm,
Step by Step.

01
Draft model generates K tokens autoregressively (fast, small model)
02
Target model scores all K+1 positions in one forward pass (parallel)
03
Accept token i if target prob ≥ draft prob or with probability ratio
04
On first rejection, resample from target and discard remaining drafts
05
Append accepted tokens. Repeat from step 01.

05 / 07

Why It
Speeds Up.

# Expected tokens per target model call
# α = acceptance rate K = draft length

E[tokens] = (1 - α^(K+1)) / (1 - α)

# Example: α = 0.8, K = 4
E[tokens] = (1 - 0.8⁵) / 0.2 ≈ 3.36 tokens

# vs standard = 1 token per call
# Speedup ≈ 3.36× same FLOPs, better util

Total FLOPs stay roughly the same. The gain is pure hardware utilization - batching verification means more tokens per memory round-trip.
