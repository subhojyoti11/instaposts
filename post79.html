What RAG
Solves.

The problem
LLMs have a training cutoff and finite knowledge. They cannot answer questions about documents added after training or internal proprietary data they have never seen. Fine-tuning on new data is expensive and slow to update.
The solution
Retrieval-Augmented Generation embeds documents into a vector database and retrieves relevant chunks at query time. Only the 3-10 most relevant chunks are inserted into the prompt. The LLM answers based on retrieved context it could not have known from training.
The cost model
RAG scales cheaply with knowledge base size. Adding 1M documents costs only embedding compute. Query cost stays fixed regardless of corpus size.
The limit
RAG retrieval is imperfect. If the relevant document is not in the top-K results, the LLM answers without it or hallucinates. Questions requiring synthesis across many documents are hard: the retriever returns local excerpts, not global understanding of the corpus.

03 / 07

What Long
Context Solves.

The context window
Gemini 1.5 Pro supports 2 million tokens. Claude 3 supports 200,000 tokens. GPT-4o supports 128,000 tokens. At 2 million tokens, you can fit an entire software codebase, a year of Slack history, or 3,000 research papers in a single prompt.
No retrieval error
If all documents fit in context, retrieval failure is impossible. The LLM attends to the full corpus directly. Questions requiring synthesis across many documents are answered accurately. No chunking strategy, no embedding model, no vector database to operate.
The cost
Attention scales quadratically with context length. A 2M Gemini query costs ~$10.50. At 100 queries/day that's $1,050 daily. RAG with a 4K prompt costs $0.004 per query.
Lost in the middle
LLMs struggle to use information from the middle of very long contexts. Stanford research showed accuracy drops for facts buried in the center of long contexts.

A RAG
Pipeline.

# RAG: cheap, scales, imperfect retrieval from openai import OpenAI import pinecone # Retrieve relevant chunks query_vec = embed("What is our refund policy?") chunks = pinecone.query(query_vec, top_k=5) # Build prompt with retrieved context only prompt = f"""Context: {chunks} Question: What is our refund policy?""" # ~4K tokens, $0.004 per query response = client.chat.completions.create( model="gpt-4o", messages=[{"role":"user", "content": prompt}] )

The Cost
Comparison.

RAG at scale
100K document corpus, 1,000 queries per day, GPT-4o at $0.004 per 4K token query. Daily cost: $4. Monthly cost: $120. Adding 100K more documents costs ~$1 one-time for embedding.
Long context at scale
Same 100K documents, 1,000 queries per day, Gemini 1.5 Pro at $10.50 per 2M token query. Daily cost: $10,500. Monthly cost: $315,000. Viable only for very infrequent, high-value queries.
Where long context wins
Low-frequency, high-value queries over complete corpora where retrieval mistakes are unacceptable. Due diligence on 300 contracts. Understanding all dependencies in a 200K line codebase. Summarizing a year of board meeting transcripts. Cost is irrelevant, completeness is required.
Hybrid
Most production systems combine both. RAG retrieves candidates. A reranker selects the top 20. Those 20 fill a 32K context for better recall than top-5 retrieval.

In
Production.

Notion AI
Uses RAG to answer questions against a user's workspace documents. Each workspace has thousands of pages. Full-context loading of an entire workspace would be prohibitively expensive per query.
Google NotebookLM
Uses long context to index a user's uploaded documents into Gemini 1.5 Pro's context window. Users upload up to 50 PDFs. NotebookLM keeps all in context, enabling cross-document synthesis.
GitHub Copilot
Uses RAG across the active codebase to provide context-aware code suggestions. Code is chunked and embedded. Long context for a large codebase would exceed 128K tokens on most models.
Anthropic Projects
Claude's Projects feature maintains persistent context across conversations using a hybrid approach. Documents are stored and selectively injected by relevance. The 200K window is used when the entire file matters.

07 / 07

Quick
Reference.

RAG
Retrieval-Augmented Generation: embed corpus into a vector database, retrieve relevant chunks at query time. Fixed query cost regardless of knowledge base size. Imperfect retrieval can miss relevant documents. Best for large, frequently-queried, frequently-updated knowledge bases.
Long context
Place the entire corpus in the model's context window for every query. No retrieval failure. Cost scales quadratically with context length. Viable for low-frequency queries where completeness matters more than cost. Gemini 1.5 Pro: 2M tokens. Claude 3: 200K tokens.
Chunking
Splitting documents into smaller segments for embedding in RAG pipelines. 256-512 tokens with 50-token overlap is the common starting point. Semantic chunking splits at sentence boundaries.
Reranking
A cross-encoder model that reranks retrieved chunks by relevance after initial vector retrieval. Retrieval finds top-50 fast. Reranker scores and keeps top-5-20. Cohere Rerank is a common choice.
Lost in middle
Stanford finding: LLM accuracy degrades for facts in the middle of long contexts. Information at the start and end of the context window is recalled reliably. Middle sections are underutilized.

Caption
When Gemini 1.5 Pro launched with a 2 million token context window, many people asked whether RAG was dead. The honest answer: not even close. RAG and long context are optimized for completely different constraints. RAG is a cost-efficiency tool. Long context is a completeness tool. RAG embeds your document corpus into a vector database, then retrieves the 3-10 most relevant chunks at query time and inserts only those into the prompt. The context window stays small. The query cost stays fixed. Adding 1 million documents to the knowledge base costs only the embedding computation. The 1,001,000th document does not increase your per-query cost by a single cent. GitHub Copilot uses RAG across codebases. Notion AI uses it across workspaces. Any product with a large, frequently-queried, frequently-updated corpus uses RAG. Long context places the entire corpus in the model's context window for every query. At 2 million tokens, there is no retrieval step. No chunking strategy. No vector database. No retrieval failure. Questions requiring synthesis across many documents, where the relevant passage could be anywhere, are answered accurately. But a 2M token Gemini query costs roughly $10.50. Run that 1,000 times per day and you are spending $315,000 per month. Google NotebookLM uses this approach for up to 50 uploaded PDFs because the users query infrequently and completeness matters. Stanford's "Lost in the Middle" paper adds a critical caveat: LLM accuracy degrades for facts in the center of long contexts. Information at the beginning and end is recalled reliably. The middle is underutilized. A 2M token context does not mean 2M tokens of uniform attention. Production systems converge on hybrid approaches: RAG retrieves candidates efficiently, a reranker selects the best 20, and those 20 fill a larger context window for better synthesis than top-5 retrieval alone.
