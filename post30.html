The KV cache problem.

> Every token in the context window creates a Key and Value vector that must be stored in GPU memory. This is the KV cache.

KV cache memory formula:
memory = 2 × layers × heads × dim × seq_len × bytes
──────────────────────────────
GPT-4o at different context lengths:
4K tokens: ~0.5 GB ← fits easily
32K tokens: ~4 GB ← noticeable
128K tokens: ~16 GB ← half a GPU
1M tokens: ~120 GB ← multiple GPUs
──────────────────────────────
PER REQUEST. Not shared. Each user
gets their own KV cache allocation.

This is why long-context is expensive. A 128K context request eats 16 GB of HBM per user. Serve 100 concurrent users and you need 1.6 TB of GPU memory. That's the memory wall in action.

Why your chat forgets.

128K context window. Chat session:
──────────────────────────────
System prompt: 2,000 tokens
Message 1-10: 15,000 tokens
Message 11-30: 45,000 tokens
Message 31-50: 60,000 tokens
Total so far: 122,000 tokens
──────────────────────────────
WARNING: approaching limit
Message 51: oldest messages DROPPED
──────────────────────────────
The model didn't "forget."
The oldest context was evicted.
Silently. Without telling you.

No LLM has persistent memory. Every "conversation" is a single prompt containing the full history. When that prompt exceeds the window, oldest messages are truncated. The model literally can't see them anymore.

Bigger ≠ better.

> "Lost in the middle" — LLMs pay most attention to the start and end of context. Information buried in the middle gets ignored.

start
First 10% of context. High attention. System prompt and initial instructions land well here
middle
Middle 60-80%. Attention drops significantly. Key facts buried here get missed or garbled
end
Last 10% of context. High attention again. The most recent message gets the most focus

1M tokens ≠ 1M tokens of understanding. You CAN stuff a whole codebase into Gemini's context. But that doesn't mean it will accurately process all of it. Retrieval quality degrades as context grows. RAG with targeted chunks often outperforms brute-force long context.

How to manage the window.

01
Summarize history. After 10 messages, compress old conversation into a 500-token summary. Reset the window
02
RAG over stuffing. Don't paste 50 pages into context. Embed and retrieve only the relevant 3-5 chunks
03
Put important info at edges. System prompt first, user query last. Don't bury instructions in the middle
04
Use prompt caching. Static system prompts cached = 50-90% cost reduction on repeated context
05
Right-size the model. Don't use 1M context model for a 4K task. Bigger windows = more compute = slower + costlier

The context window is your budget. Spend it wisely. Every token inside competes for attention. Smaller, focused context often produces better results than massive, unfocused context. Quality of context > quantity of context.
